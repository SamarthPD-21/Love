# Notification System + Maximal Fun — Implementation Plan

Stack stays the same (Express + Mongoose, Next 16 + React Query + Zustand + Framer Motion). Two new deps only: `socket.io` (+types) on the server, `socket.io-client` on the client. JWT reused for socket auth.

---

## PART A — Notification System (instant, via WebSockets)

### A1. New Mongoose model — `server/src/models/Notification.ts`
Fields (mirrors existing ownership pattern `relationshipId` + `userId`):
- `relationshipId` (ref Relationship, indexed)
- `actorUserId` (ref User — who did it)
- `recipientUserId` (ref User — the partner)
- `type` — string enum of event keys (see A3)
- `entityType` (string, e.g. "Letter"), `entityId` (ObjectId, polymorphic — not a strict ref so it survives deletions)
- `title` — pre-rendered human string with emoji (e.g. "Aisha wrote you a letter 💌")
- `isRead` (Boolean, default false)
- `createdAt`

### A2. Socket.io server — `server/src/sockets.ts`
- Initialize `http.Server` + `io` in `index.ts` (replace `app.listen` with `server.listen`).
- Auth middleware: verify JWT from `socket.handshake.auth.token` (same secret/logic as `middleware/auth.ts`); attach `socket.userId`. Disconnect unauth.
- On connect: `socket.join("user:<userId>")`.
- Export an `emitToUser(userId, event, payload)` helper used by the notify service.
- Configure CORS for the client origin.

### A3. Notify helper — `server/src/services/notify.ts`
`createNotification({ actorId, type, entityType?, entityId?, meta? })`:
1. Load actor → get `relationshipId` + `partnerId` (recipient). No partner → no-op.
2. Build the warm/playful `title` from a config map keyed by `type` (e.g. `hug_sent` → "You got a hug! 🫂💕", `letter_created` → "{name} wrote you a letter 💌", `memory_deleted` → "{name} removed a memory 🗑️"). Substitutes actor name + meta.
3. Insert Notification doc.
4. `emitToUser(recipientId, "notification", notification)` for instant push.
5. Fully guarded (try/catch, non-fatal) so a notify failure never breaks the originating request.

Supported `type` values: `letter_created/deleted`, `memory_created/deleted`, `memory_commented`, `voice_created`, `jar_note_created`, `hug_sent`, `countdown_created`, `gratitude_added`, `milestone_added`, `song_added`, `map_pin_added`, `profile_updated`, `letter_unlocked`.

### A4. Notifications REST route — `server/src/routes/notifications.ts` (mounted in `index.ts`)
- `GET /api/notifications` — list newest-first, `?limit=&before=` cursor, populates actor (name/avatar).
- `GET /api/notifications/unread-count` → `{ count }`.
- `PATCH /api/notifications/read` — mark all read.
- `PATCH /api/notifications/:id/read` — mark one.
- `DELETE /api/notifications/:id` — dismiss.

### A5. Instrument routes (~2 lines each at success points)
`letters.ts` (POST create, DELETE, POST /:id/unlock), `memories.ts` (POST, DELETE, comment add), `voice-notes.ts` (POST), `memory-jar.ts` (POST), `users.ts` (POST /hugs, PUT /me), `countdowns.ts`, `gratitude.ts`, `timeline.ts`, `songs.ts`, `map.ts` (POST create events). All fire-and-forget `createNotification(...)`.

## PART B — Frontend notifications

### B1. Types — `client/types/index.ts`
Add `AppNotification` interface matching the model.

### B2. Socket client — `client/lib/socket.ts`
Singleton `io(NEXT_PUBLIC_SOCKET_URL || origin-of-API, { auth: { token } })`. Reconnect, re-auth on token change, disconnect on logout.

### B3. Hook — `client/hooks/useNotifications.ts`
- `useUnreadCount()` — `useQuery(["notifications","unread"], GET unread-count)` for initial/hydration value (kept fresh by socket events invalidating it).
- `useNotifications()` — list query + infinite scroll via cursor.
- Mutations: `useMarkAllRead`, `useMarkOneRead`, `useDismiss`.
- `useLiveNotifications()` — opens the socket, subscribes to `"notification"`; on new event: invalidates unread + list, fires `playSound("notification")`, pushes a celebratory toast (via `useToastStore`), and triggers confetti for "big" types.

### B4. `NotificationBell` + dropdown — `client/components/notifications/NotificationBell.tsx`
- Animated bell (lucide `Bell`) with Framer Motion unread badge (heart-shaped, pop/scale), wiggle when count rises.
- Dropdown panel (glassmorphic, `AnimatePresence`): actor avatar, title, relative time (`date-fns formatDistanceToNow`), unread dot, "Mark all read", deep-link to relevant page (entityType → href map), empty state.
- Reused in `Sidebar.tsx` (desktop footer) AND `AppShell.tsx` mobile header.

### B5. Global wiring
- Mount `useLiveNotifications()` once (in `AppShell` so it's active on every protected screen) and render `<NotificationBell/>` in both nav surfaces. Connect socket after auth is ready (token available).

## PART C — Maximal Fun ✨

### C1. Confetti — `client/components/animations/Confetti.tsx`
Lightweight canvas burst (no dep). Driven by a `useCelebration` hook backed by a Zustand `useCelebrationStore` (`celebrate(intensity)`). Triggered on: hug sent, surprise opened, letter unlocked, countdown reached, and incoming "big" notifications.

### C2. Ambient `SparkleTrail.tsx`
New ambient layer alongside `FloatingHearts`/`Fireflies` — drifting sparkles; render in `AppShell` (day) and a softer variant at night.

### C3. Living home dashboard — `app/(main)/page.tsx`
- New **"Recent Activity"** strip showing latest 4 notifications (reuses feed). Added between hero and shortcuts.
- Parallax drift on hero glows (mouse/scroll-tied Framer Motion).

### C4. Reaction emojis — `client/components/animations/FloatingReaction.tsx`
Triggered floating-emoji burst (hearts/sparkles) from a tap point — used on hug, memory favorite, and notification bell open.

### C5. Micro-interaction polish
- Wiggle/bounce on emoji shortcuts & partner avatar hover.
- Bell open → `pop` sound; new notification → `notification` sound + confetti.
- Extended toast: add a `celebration` variant to `useToastStore`/`ToastContainer` (bigger, emoji, gradient).
- Full-screen celebratory moment for milestone/anniversary-style events.

### C6. Playful copy
The notify title map in A3 carries the warm personality (emojis, partner's name) — the "fun" voice lives server-side so it's consistent.

## Implementation order
1. Install deps (`socket.io`+types server; `socket.io-client` client).
2. Notification model.
3. `sockets.ts` + rewrite `index.ts` to use `http.Server` + mount `/notifications`.
4. `services/notify.ts` (with copy/emoji map).
5. Notifications REST route.
6. Instrument ~10 routes.
7. Client: types, `lib/socket.ts`, `useNotifications` hook.
8. `Confetti` + `useCelebration` + extend toast store; `SparkleTrail`, `FloatingReaction`.
9. `NotificationBell` + dropdown.
10. Wire bell + `useLiveNotifications` into `AppShell`/`Sidebar`; add "Recent Activity" to home; ambient + micro-interactions.
11. **Build-check**: `tsc --noEmit` on server, `next build`/`tsc` on client; fix errors.

## Constraints / what I will NOT change
- No changes to auth/invite flow, existing CRUD behavior, or DB schemas beyond the new Notification collection.
- Socket auth reuses the existing JWT secret — no separate auth system.
- All new code matches existing conventions (`res.json({success,data})`, `relationshipId` scoping, `authMiddleware`, glassmorphic Tailwind, Framer Motion variants, `useSoundEffects`/`useToastStore`).
- Two new deps only; no other infra.
